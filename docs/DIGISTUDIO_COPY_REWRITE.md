# Digistudio 文案去噪与重构

## 目标
这次调整解决的不是“文案不够高级”，而是另外两个更实际的问题：

1. 术语太多，读者要先翻译一遍，理解成本高。
2. 话说得很满，但真正的用户痛点被盖住了。

文案原则统一成一句话：

`用大白话讲清人话，用专业词提炼方法。`

也就是说：

- 讲问题时，说用户到底卡在哪。
- 讲方案时，说你具体改了什么。
- 讲方法时，再用一层专业表达做总结。

## 去噪翻译
下面是当前项目里最容易显得“包装感过强”的说法，以及更自然的替换方向。

| 原表达 | 问题 | 去噪翻译 |
| --- | --- | --- |
| 可视化画布重构 | 太像汇报标题，不像人在说话 | 把复杂对话改成一张能看懂、能继续改的画布 |
| 可视化画布生命周期 | 抽象过头，用户感知不到 | 先把剧情关系摆出来，再让配置跟上 |
| 双轨创建矩阵 | 像方法论名词，不像功能 | 两种创建方式：顺手往下接 / 先搭骨架 |
| 高密网络下的视觉降噪 | 学术味重，面试里不够落地 | 关系再复杂，也要一眼能扫明白 |
| 验证产能 | 过于平台化、汇报化 | 看真实结果 / 看它是不是真的更快 |
| 结构外显 | 表达对，但太像术语 | 先把关系摆出来 |
| 边界防守 | 含义模糊 | 明显会出错的地方，直接拦住 |
| 信任模型优先于自动聪明 | 句式很“策略”，但不够人话 | 删节点时，我宁愿让用户自己接，也不替他乱接 |
| 逻辑盲视 | 生造词感较重 | 回头改的时候，根本找不到刚才那段逻辑 |
| 整体产能提升 | 太像业务周报 | 大家做得更快，也更愿意继续改下去 |

## 已落到页面的成稿方向
当前 [digistudio.html](/Users/yezehong/Documents/Codex/2026-05-31/files-mentioned-by-the-user-project/portfolio-site/digistudio.html) 已经切换到一版“自然但仍专业”的成稿，特点是：

- 标题从“方法论命名”改成“结果导向表达”。
- 每一段都优先说明用户卡点，再说设计动作。
- 第一人称只放在关键判断和取舍处，不铺满整页。
- 标题和小标题尽量压短，避免整页都是“口播感文案”。

这一版适合：

- 作品集正式上线
- 面试官快速浏览
- 需要显得成熟，但不能太端着的场景

## 更像面试讲述的全文案重构方案
如果你希望它更像“我怎么发现问题、怎么下手解决、最后验证有效”的面试讲述，重点不是再口语一点，而是把下面三件事写出来：

- 我是怎么发现这不是表层问题的。
- 我做了哪些关键判断和取舍。
- 我最后怎么验证这些判断是对的。

下面是整页可替换的“面试讲述版”文案。

### Hero

**标题**

`Digistudio UGC 编辑器：把“到处找配置”改成“看着一张图就能继续做”`

**导语**

我接手这个项目时，很快就发现它的问题不是界面旧，而是思路和工具完全对不上。  
会写复杂剧情的人，脑子里想的是一张分支关系图，但旧工具给他的，却是一堆分散在不同面板里的配置项。  
所以我给自己定的目标不是“把界面做得更像编辑器”，而是先把这些被拆散的关系重新放回一张画布里，让人能看着它继续写、继续改，也能回头找到刚才那段逻辑。

### Process

**01 我先确认问题**

创作者不是不会写，而是工具把他们的思路打断了。

**02 我先搭对结构**

先把剧情关系摆出来，再谈配置效率。

**03 我再补安全感**

该拦的错误提前拦，该看见的信息直接露出来。

**04 最后我看结果**

不是“更像专业工具”，而是真的更快、更敢改。

### Problem

**标题**

`剧情复杂不是问题，关系被拆散才是问题`

**正文**

我一开始先去看，真正卡住的是谁、卡在什么地方。  
这类 UGC 创作者要处理的不只是对白，还有任务状态、好感度分支、条件判断和功能触发。  
他们脑子里天然是一张关系图，但旧系统把这些内容拆在不同组件和不同面板里。  
一旦关系看不见，创作者就会把大量时间花在“找”和“确认”上，而不是花在真正的创作上。

**三条痛点**

- 一段完整对话要拆开配，流程很碎。
- 文本一多就很难回看，改起来更痛苦。
- 分支和跳转关系藏起来以后，只能靠记忆硬撑。

**专业提炼**

参考 UE 蓝图、Roblox Studio 这类成熟编辑器后，我确认了一件事：  
复杂逻辑不能继续靠堆表单解决，必须先把关系放到台面上。

### Journey

**标题**

`创作者最崩溃的，不是剧情难写，而是越写越找不到自己刚配的东西`

**导语**

为了判断问题到底卡在哪一段，我把“做一个带好感度分支的任务”拆成完整过程看了一遍。最后我发现，问题并不发生在构思阶段，而是发生在“开始把想法落进工具”的那一刻。

**四步叙事**

1. 构思时，创作者脑中的关系其实很清楚。
2. 一开始搭逻辑，工具就已经跟不上脑中的分支图。
3. 进入具体配置后，人会在不同面板之间频繁跳转，越配越乱。
4. 到了测试和回改阶段，很多人已经找不到刚才那段逻辑到底藏在哪。

### Phase 01

**标题**

`第一步：先让人把剧情关系摆出来，再让配置跟上`

**正文**

当时我先做了一个判断：如果关系本身看不见，后面无论把表单打磨得多精致，用户还是会迷路。  
所以我没有直接把原来的文本框搬到画布上，而是先做了一张真正能表达剧情关系的对话图。  
节点代表一句话、一个动作或一个状态，线代表它们之间的走向；文本、条件和功能都直接挂在节点上。  
这样创作者不需要在脑中反复拼图，而是能直接看着关系继续往下做。

**两种创建方式**

- `顺手往下接`
  这是我优先照顾的高频路径。拉一根线，松手就能接着建下一个节点。
- `先搭骨架`
  这是我给规划型用户留的路径。先把主要分支摆出来，再慢慢补内容和细节。

### Phase 02

**标题**

`第二步：别让系统自作聪明，也别让复杂关系把人看晕`

**正文**

做到这一步以后，我没有急着继续加功能，而是先处理另外一件更关键的事：用户敢不敢信这个系统。  
画布能用之后，真正决定体验好不好的，是“用户敢不敢放心用”。  
所以第二阶段我重点处理三件事：

- 系统什么时候该帮，什么时候不能乱帮。
- 明显会出错的地方，能不能在当下就拦住。
- 信息一多以后，用户还能不能快速扫懂画面。

**三个关键决策**

- 删节点时，不做自动乱接。这是我专门做过取舍的地方，复杂逻辑里，可控比省一步更重要。
- 连线形成死循环时，当场拦截。这是我主动往前提的防线，不把坑留到测试阶段。
- 条件、功能、分组这些关键信息直接露出来。这里我的判断是，复杂不是问题，看不清才是问题。

### Impact

**标题**

`最后看结果：不是更好看了，而是真的省时间`

**正文**

为了避免只是在“自己感觉更顺”，我找了 4 名内部 UGC 创作者，用新旧两套系统分别完成同一个中大型剧情任务。  
任务里包含 3 个好感度分支和 4 个复合功能触发。  
我们记录他们找节点、配置内容和整体完成时间，验证这次改版到底有没有把时间从“反复找配置”里省出来。

**结果表达**

- `< 1s`
  想找哪段逻辑，几乎立刻就能看到。
- `1h → 15m`
  同一任务的配置时间明显缩短。
- `+80%`
  更重要的是，大家终于愿意继续改、继续扩，因为改动成本降下来了。

### Closing

**标题**

`这个项目最后证明的，其实很简单`

**正文**

这次项目最后让我更确定一件事：复杂编辑器最重要的，不是功能堆得多，而是用户每走一步都知道自己在哪、刚刚改了什么、接下来会影响哪里。

**收束句**

我现在会把好的专业工具理解成这样：它不是替人做决定，而是在复杂决定里把信息摆清楚，把风险拦在前面。  
对我来说，这次 Digistudio 重构最关键的收获，不是界面更像编辑器了，而是我把“关系先被看见，效率才会真正提升”这件事做实了。

## 使用建议

如果后面继续调文案，我建议这样分层：

- 首页卡片：一句话讲结果，别讲方法论。
- 项目页标题：讲“解决了什么问题”，不要先讲“采用了什么框架”。
- 正文段落：先说用户卡点，再说你怎么改。
- 面试讲述：先讲自己怎么判断，再讲自己做了什么。

一句话判断标准：

`如果一个不做这个项目的人也能一下读懂，那这版文案就对了。`
